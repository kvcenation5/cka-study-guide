    Question 1: Create StorageClass with Default Setting

    Task: Create a StorageClass named local-sc with the following specifications and set it as the default storage class:

    • Provisioner: kubernetes.io/no-provisioner
    • Volume binding mode: WaitForFirstConsumer
    • Volume expansion: enabled
    • Mark as default storage class

    Answer: 

    # Create the StorageClass
kubectl create -f - <<EOF
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-sc
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
EOF

# Verification commands
kubectl get storageclass local-sc
kubectl describe storageclass local-sc
kubectl get storageclass  # Check for (default) marker
kubectl get storageclass local-sc -o yaml | grep -E "(provisioner|volumeBindingMode|allowVolumeExpansion|is-default-class)"

    Verification Checklist: - ✅ StorageClass local-sc created - ✅ Provisioner is kubernetes.io/no-provisioner - ✅ Volume binding mode is WaitForFirstConsumer - ✅ Volume expansion is enabled (allowVolumeExpansion: true) - ✅ Set as default storage class

    Question 2: Deployment with Sidecar Container

    Task: Create a deployment named logging-deployment in the namespace logging-ns with 1 replica, with the following specifications:

    Main Container (app-container): - Image: busybox - Command: sh -c "while true; do echo 'Log entry' >> /var/log/app/app.log; sleep 5; done"

    Sidecar Container (log-agent): - Image: busybox - Command: tail -f /var/log/app/app.log - Must read logs from main container

    Requirements: - Use a shared emptyDir volume for log sharing - Both containers must mount the same volume at /var/log/app - Sidecar should display logs from main container in real-time

    Answer: 

    # Create namespace first
kubectl create namespace logging-ns

# Create the deployment
cat > logging-deployment.yaml << EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: logging-deployment
  namespace: logging-ns
spec:
  replicas: 1
  selector:
    matchLabels:
      app: logging
  template:
    metadata:
      labels:
        app: logging
    spec:
      containers:
      - name: app-container
        image: busybox
        command: ["sh", "-c", "while true; do echo 'Log entry' >> /var/log/app/app.log; sleep 5; done"]
        volumeMounts:
        - name: log-volume
          mountPath: /var/log/app
      - name: log-agent
        image: busybox
        command: ["tail", "-f", "/var/log/app/app.log"]
        volumeMounts:
        - name: log-volume
          mountPath: /var/log/app
      volumes:
      - name: log-volume
        emptyDir: {}
EOF

kubectl apply -f logging-deployment.yaml

# Verification commands
kubectl get deployment logging-deployment -n logging-ns
kubectl get pods -n logging-ns
kubectl logs -n logging-ns <pod-name> -c log-agent
kubectl describe pod -n logging-ns <pod-name>

    Verification Checklist: - ✅ Deployment logging-deployment created in logging-ns namespace - ✅ Main container (app-container) writes logs to shared volume - ✅ Sidecar container (log-agent) reads logs from same volume - ✅ Shared emptyDir volume mounted at /var/log/app in both containers - ✅ Sidecar properly displays logs from main container

    Question 3: View Sidecar Container Logs

    Scenario: You have a deployment with multiple containers (main app and sidecar). You need to view logs from a specific container.

    Given: A pod logging-deployment-59df466f64-56299 in namespace logging-ns with two containers: - app-container (main application) - log-agent (sidecar for log aggregation)

    Task: View the logs from the log-agent sidecar container specifically.

    Answer: 

    # View logs from specific sidecar container
kubectl logs logging-deployment-59df466f64-56299 -n logging-ns -c log-agent

# Follow logs in real-time (like tail -f)
kubectl logs logging-deployment-59df466f64-56299 -n logging-ns -c log-agent -f

# View logs from all containers
kubectl logs logging-deployment-59df466f64-56299 -n logging-ns --all-containers=true

# View logs from default container (first one)
kubectl logs logging-deployment-59df466f64-56299 -n logging-ns
# Note: This defaults to "app-container" and shows message:
# "Defaulted container 'app-container' out of: app-container, log-agent"

    Key Commands: 

    # Single container logs (must specify -c if not default)
kubectl logs <pod-name> -n <namespace> -c <container-name>

# All containers
kubectl logs <pod-name> -n <namespace> --all-containers=true

# Follow/tail logs
kubectl logs <pod-name> -n <namespace> -c <container-name> -f

# Previous container logs (if crashed/restarted)
kubectl logs <pod-name> -n <namespace> -c <container-name> --previous

    Verification Checklist: - ✅ Can view logs from specific sidecar container using -c flag - ✅ Understands default container selection behavior - ✅ Knows how to view all containers' logs - ✅ Can follow logs in real-time with -f flag

    Question 4: Create Ingress Resource

    Scenario: A Deployment named webapp-deploy is running in the ingress-ns namespace and is exposed via a Service named webapp-svc.

    Task: Create an Ingress resource called webapp-ingress in the same namespace that will route traffic to the service.

    Requirements: - Use pathType: Prefix - Route requests sent to path / to the backend service - Forward traffic to port 80 of the service - Be configured for the host kodekloud-ingress.app - Add ingress class annotation: kubernetes.io/ingress.class: nginx

    Test Command: 

    curl -s http://kodekloud-ingress.app/

    Answer:

    Method 1: Imperative Command (Quick) 

    # Create Ingress using kubectl create (uses ImplementationSpecific pathType by default)
kubectl create ingress webapp-ingress --class=nginx -n ingress-ns --rule="kodekloud-ingress.app/=webapp-svc:80"

# Note: To set exact pathType: Prefix, edit the ingress after creation
kubectl edit ingress webapp-ingress -n ingress-ns
# Change: pathType: ImplementationSpecific → pathType: Prefix

    Method 2: Declarative YAML (Exact Specifications) 

    # Create the Ingress with exact requirements
kubectl create -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: webapp-ingress
  namespace: ingress-ns
  annotations:
    kubernetes.io/ingress.class: "nginx"
spec:
  rules:
  - host: kodekloud-ingress.app
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: webapp-svc
            port:
              number: 80
EOF

# Verification commands
kubectl get ingress webapp-ingress -n ingress-ns
kubectl describe ingress webapp-ingress -n ingress-ns
kubectl get ingress webapp-ingress -n ingress-ns -o yaml

# Test the Ingress
curl -s http://kodekloud-ingress.app/

    Troubleshooting (if you get 404): 

    # Check if ingress controller is running
kubectl get pods -n ingress-nginx

# Check service endpoints
kubectl get endpoints webapp-svc -n ingress-ns

# Test service directly
kubectl get svc webapp-svc -n ingress-ns

# Add ingress class annotation if missing
kubectl annotate ingress webapp-ingress -n ingress-ns kubernetes.io/ingress.class=nginx

    Verification Checklist: - ✅ Ingress webapp-ingress created in ingress-ns namespace - ✅ pathType set to Prefix - ✅ Backend service webapp-svc on port 80 - ✅ Host configured as kodekloud-ingress.app - ✅ Ingress class annotation added (kubernetes.io/ingress.class: nginx) - ✅ Ingress exposed and serving traffic via kodekloud-ingress.app host

    Question 5: Create User with CSR and RBAC Role

    Scenario: Create a new user "john" with certificate-based authentication and grant specific permissions.

    Given: - Private key: /root/CKA/john.key - CSR file: /root/CKA/john.csr

    Task: 1. Create a CertificateSigningRequest (CSR) named john-developer for user john 2. Approve the CSR 3. Create a Role named developer in development namespace 4. Grant permissions: create, list, get, update, delete pods 5. Create RoleBinding to bind john to the developer role

    Answer: 

    # Step 1: Create the CSR
# Read the CSR file and encode it in base64
CSR=$(cat /root/CKA/john.csr | base64 | tr -d '\n')

# Create the CertificateSigningRequest
kubectl create -f - <<EOF
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: john-developer
spec:
  request: $CSR
  signerName: kubernetes.io/kube-apiserver-client
  usages:
  - client auth
  expirationSeconds: 86400
EOF

# Step 2: Approve the CSR
kubectl certificate approve john-developer

# Step 3: Extract the signed certificate
kubectl get csr john-developer -o jsonpath='{.status.certificate}' | base64 -d > /root/CKA/john.crt

# Step 4: Create the developer role
cat > developer-role.yaml << EOF
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
  namespace: development
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["create", "list", "get", "update", "delete"]
EOF

kubectl apply -f developer-role.yaml

# Step 5: Create RoleBinding for john
cat > john-binding.yaml << EOF
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: john-developer-binding
  namespace: development
subjects:
- kind: User
  name: john
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io
EOF

kubectl apply -f john-binding.yaml

# Step 6: Configure kubectl for john
kubectl config set-credentials john --client-certificate=/root/CKA/john.crt --client-key=/root/CKA/john.key
kubectl config set-context john-context --cluster=$(kubectl config current-context | cut -d/ -f2) --namespace=development --user=john

    Verification Commands: 

    # Check CSR status
kubectl get csr john-developer

# Check Role
kubectl get role developer -n development
kubectl describe role developer -n development

# Check RoleBinding
kubectl get rolebinding john-developer-binding -n development

# Test as john user
kubectl --context=john-context auth can-i create pods -n development
kubectl --context=john-context auth can-i list pods -n development
kubectl --context=john-context auth can-i delete pods -n development

# View current context
kubectl config get-contexts

# Switch to john context
kubectl config use-context john-context

# Try creating a pod as john
kubectl --context=john-context run test-pod --image=nginx -n development --dry-run=client

    Verification Checklist: - ✅ CSR john-developer created and Approved - ✅ Role developer created in development namespace - ✅ Role has permissions: create, list, get, update, delete pods - ✅ RoleBinding connects user john to developer role - ✅ User john has appropriate permissions in development namespace
